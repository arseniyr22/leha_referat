# DECISIONS LOG — Все ключевые решения проекта

Хронологический лог решений, принятых за все сессии. Каждое решение — финальное, не пересматривать без явного запроса.

## Архитектурные решения

### D1: Wrap, не rewrite
**Решение**: Существующий pipeline (10 модулей, 4806 строк) оборачивается в агентную систему. Код pipeline/ НЕ переписывается.
**Причина**: Сохраняет натренированные промпты, 150+ тестов, config.yaml. Потеря при rewrite > выигрыш.
**Дата**: Сессия 2

### D2: Custom orchestrator на anthropic SDK
**Решение**: НЕ использовать LangGraph, CrewAI, AutoGen. Писать оркестратор самостоятельно на anthropic SDK.
**Причина**: LangGraph/CrewAI добавляют лишнюю абстракцию, ломают explicit state passing, усложняют отладку.
**Дата**: Сессия 2

### D3: Архитектура v3.1 FINAL (82 компонента)
**Решение**: 4-уровневая иерархия: CEO → Manager (2) → Micro Manager (7) → Worker (36 content + 19 HM) + Gates (5) + Services (7) + Export (1)
**Причина**: v1 (flat) слишком простая, v2 (3-level) не покрывала MM, v3 не имела feedback loop. v3.1 добавила Smart Worker Skip + Cost Tracking.
**Дата**: Сессия 2-3

### D4: Два режима работы
**Решение**: Generation mode (topic → text → humanize → docx) и Humanization mode (text → humanize → docx)
**Причина**: SaaS нужны оба — генерация с нуля и обработка готового текста
**Дата**: Сессия 1

### D5: Single-generation streams через MM
**Решение**: text/essay/composition проходят через MicroManager с SECTION_ORDER=["full"], вызывают generator.generate() (не generate_section())
**Причина**: Единообразие интерфейса. ContentManager всегда вызывает mm.execute(state).
**Дата**: Сессия 4

## Решения по скиллам

### D6: 8 кастомных скиллов (не внешние)
**Решение**: Создавать свои скиллы вместо скачивания из GitHub-репозиториев.
**Причина**: Проанализированы 3 репозитория (everything-claude-code: 65 скиллов, awesome-claude-skills: 27, blader/humanizer: 1). Вывод: generic скиллы не дают value сверх CLAUDE.md + существующего кода. Наша система слишком специфична.
**Дата**: Сессия 3-4

### D7: Скиллы — для BUILD, не для RUNTIME
**Решение**: Скиллы помогают при РАЗРАБОТКЕ (генерация boilerplate, написание промптов, отладка). Агенты в RUNTIME используют свои промпты из agents/prompts/, не скиллы.
**Причина**: Скиллы = Claude Code context. Агентные промпты = anthropic SDK runtime. Разные системы.
**Исключения**: academic-writer и academic-visualizer — BUILD+RUNTIME (улучшают и промпты при разработке, и потенциально содержание при генерации)
**Дата**: Сессия 3

### D8: 6-Step Skill Building Framework
**Решение**: Все скиллы создаются по фреймворку: (1) Name+Trigger, (2) Goal, (3) Step-by-Step Process, (4) Reference Files, (5) Rules, (6) Self-Improvement
**Причина**: Структурированный подход. Предыдущая версия промпта покрывала только шаги 3 и 5.
**Дата**: Сессия 4

### D9: cost-aware-llm-pipeline — тест перед принятием
**Решение**: Скачать, протестировать по 3 критериям (релевантность, конфликты, ROI). Если 2/3 = да → оставить. Иначе → удалить.
**Причина**: Не уверены в ценности — Appendix B уже покрывает большинство оптимизаций.
**Дата**: Сессия 4

### D10: humanizer — отладочный инструмент, не dependency
**Решение**: git clone для ручного тестирования. НЕ production dependency.
**Причина**: Наша система (53 паттерна + Russian + GOST + domain rules) превосходит humanizer (24 паттерна, English only).
**Дата**: Сессия 3

## Решения по стоимости

### D11: Prompt Caching обязателен
**Решение**: cache_control=ephemeral на все system prompts. Экономия ~$2.70 (−27%) за ВКР.
**Причина**: Megaprompt (12K tok) × 9 секций = 8 cache hits. Stage prompts (2.5K tok) × 83 чанка = 82 cache hits.
**Дата**: Сессия 4

### D12: Smart Worker Skip
**Решение**: HM-1 анализ определяет какие workers нужны. Если паттернов 0 → skip worker. Экономия 15-30%.
**Причина**: Не все тексты содержат все паттерны. Зачем вызывать Triplet Buster если триплетов нет?
**Дата**: Сессия 4

### D13: Batch API для эконом-тарифа
**Решение**: Опциональный режим через anthropic.Batch.create(). 50% скидка, результат через 1-2 часа.
**Причина**: Для SaaS можно предложить два тарифа: "быстро" (real-time) и "эконом" (batch).
**Дата**: Сессия 4

### D14: Рекомендуемая цена SaaS
**Решение**: 1,500–2,000₽ за ВКР бакалавр. Себестоимость ~650₽ ($7.20 с кэшем). Маржа 57-68%.
**Причина**: Безубыточность при ~700₽. Ниже 1,000₽ — слишком низкая маржа.
**Дата**: Сессия 4

## Решения по скиллам (обновления из анализа транскриптов)

### D17: Feedback cycle обязателен для каждого скилла
**Решение**: Каждый скилл после создания проходит цикл: invoke → watch behavior → give feedback → fix SKILL.md → repeat (минимум 2-3 итерации).
**Причина**: Транскрипт "Master 95% of Claude Code Skills" показывает, что скиллы без feedback cycle работают на 40-60% от потенциала. Self-Improvement (шаг 6 фреймворка) — это не просто лог, а активный цикл тестирования.
**Дата**: Сессия 5

### D18: Front matter опции для контроля скиллов
**Решение**: Использовать YAML front matter опции в скиллах:
- `allowed_tools` — ограничивать инструменты (pattern-auditor = только Read/Grep, не Edit/Write)
- `model` — форсировать модель (Haiku для дешёвых операций, Opus для сложных)
- `agent: true` — запускать как sub-agent с изолированным контекстом (для pattern-auditor и ac-gate)
**Причина**: Экономия токенов, изоляция контекста, предотвращение случайных изменений файлов аналитическими скиллами.
**Дата**: Сессия 5

### D19: Хардкодинг известных значений в скиллах
**Решение**: Вместо runtime-поиска файлов/путей/конфигов — хардкодить в SKILL.md или reference/ всё, что известно заранее:
- agent-builder: пути к pipeline/*.py, типы агентов, все placeholders
- prompt-crafter: все 38 принципов, все ban-листы
- ac-gate: все 15 AC-правил
- pattern-auditor: все P1-P53 паттерны
**Причина**: Каждый поиск файла = потерянные токены. Хардкод экономит 200-500 токенов на вызов.
**Дата**: Сессия 5

### D20: Global vs Local размещение скиллов
**Решение**:
- LOCAL (`.claude/skills/` в проекте): agent-builder, prompt-crafter, academic-writer, academic-visualizer, ac-gate, pattern-auditor, cost-aware-llm-pipeline — все project-specific
- GLOBAL (`~/.claude/skills/`): humanizer — универсальный инструмент для любого проекта
**Причина**: 7 из 8 скиллов специфичны для AI Anti-Anti Plag. Только humanizer — generic.
**Дата**: Сессия 5

### D21: agent-builder размещение — LOCAL, не GLOBAL
**Решение**: Перенести agent-builder из `~/.claude/skills/` в `.claude/skills/` (в проекте).
**Причина**: agent-builder использует шаблоны и правила специфичные для нашего проекта (PipelineState, AC-gates, R1-R12). В другом проекте он бесполезен.
**Дата**: Сессия 5

## Решения по плагинам

### D22: context7 — установить сейчас
**Решение**: Установить плагин context7 (Upstash Context7 MCP). Используется с Phase 0+.
**Причина**: Подтягивает актуальную документацию библиотек (python-docx, spaCy, anthropic SDK, transformers) прямо в контекст Claude Code. Риск = 0, конфликтов с нашими скиллами нет. Экономит время когда Claude Code пишет код с API.
**Когда использовать в промптах**: Указывать Claude Code "используй context7 для проверки API python-docx/spaCy/anthropic" когда промпт касается написания кода с внешними библиотеками.
**Дата**: Сессия 6

### D23: superpowers — установить на Phase 2+
**Решение**: НЕ устанавливать сейчас. Установить после завершения Pre-Build, на Phase 2+ когда начнётся массовое написание кода (36 content workers + 19 HM workers).
**Причина**: Сейчас risk конфликта с нашими скиллами (agent-builder управляет процессом создания кода; superpowers тоже управляет процессом — два "начальника" = путаница). На Phase 2+ наши скиллы генерируют boilerplate, а superpowers может обеспечить TDD discipline и debugging при доработке. К тому моменту workflow устоится, и конфликт маловероятен.
**Триггер установки**: Когда первые 5+ workers написаны и требуется систематический debugging/testing.
**Дата**: Сессия 6

### D24: code-review — установить на Phase 3+
**Решение**: Установить когда кодовая база достигнет ~20+ Python-файлов (Phase 3+).
**Причина**: Сейчас кода pipeline нет (Pre-Build фаза — только скиллы). Code review бесполезен без кода.
**Дата**: Сессия 6

### D25: pyright-lsp — установить сейчас
**Решение**: Установить плагин pyright-lsp (global). Используется с Phase 0+.
**Причина**: Статический type checking для Python. Ловит ошибки типов при передаче PipelineState между 82 компонентами. Все function signatures имеют type hints по конвенции CLAUDE.md.
**Дата**: Сессия 6

## Решения по качеству источников

### D26: Source Verification Rules — спецификация для source_finder worker
**Решение**: Не создавать отдельный скилл. Вместо этого — зафиксировать единый набор правил верификации источников, который source_finder worker (Phase 2) обязан реализовать. Правила добавляются в CLAUDE.md секцию Phase 0A.
**Причина**: Правила верификации сейчас размазаны между CLAUDE.md (Phase 0A: 3 слоя), academic_megaprompt.md (БЛОК 8: ГОСТ, категории, приоритеты), и config.yaml (source_minimums). Конкурент marsik.ai имеет отдельного "Библиографического агента" для свежести/точности. У нас функция покрыта source_finder.py, но правила верификации не формализованы.

**Правила верификации (для реализации в source_finder worker, Phase 2):**

**1. Свежесть источников:**
- Основной массив (≥70% списка): публикации последних 5 лет (2021+)
- Допустимо (≤20%): классические/фундаментальные работы (более 5 лет) — с пометкой "фундаментальный"
- Обязательно (≥10%): публикации последнего года (2025-2026) — демонстрирует актуальность
- Для `law`: НПА проверяются на действующую редакцию (не утратили силу)
- Для `it_cs`: технологические источники старше 3 лет — флагировать как потенциально устаревшие

**2. Точность (валидация):**
- Layer 2 (Semantic Scholar API): fuzzy match threshold 0.85 на title+author+year
- Дополнительно (если Semantic Scholar не нашёл): CrossRef API как fallback
- Для российских источников: проверка через eLibrary.ru РИНЦ API (если доступен)
- DOI-резолюция: если DOI найден — проверить что он резолвится (не 404)
- Источники не прошедшие валидацию: маркировать `[NEEDS VERIFICATION]` + включать в score_report.sources.needs_verification_list

**3. Качество источникового пула:**
- Минимум 2 типа источников (не все из одной категории: не все статьи, не все учебники)
- Для `vkr`/`coursework`/`research`: минимум 1 иностранный источник
- Для `law`: минимум 3 НПА + минимум 3 судебных решения с реквизитами
- Для `psychology`: минимум 2 peer-reviewed journal sources
- Для `economics`: минимум 1 источник Росстат/ЦБ/международная организация
- Нулевая толерантность к: Википедии как научному источнику, рефератным сайтам, источникам без автора/даты

**4. ГОСТ-комплаенс (Р 7.0.100-2018):**
- Все источники форматируются строго по ГОСТ (уже реализовано в source_finder.py)
- Категоризация в правильном порядке (НПА → монографии → статьи → электронные)
- Электронные ресурсы: обязательно URL + "дата обращения: ДД.ММ.ГГГГ"

**5. Score report расширение:**
```json
{
  "sources": {
    "total": 22,
    "freshness": {
      "last_year": 4, "last_5_years": 15, "older": 3,
      "freshness_ratio": 0.86, "pass": true
    },
    "verification": {
      "verified_semantic_scholar": 16, "verified_crossref": 2,
      "needs_verification": 4,
      "needs_verification_list": ["Source N — title mismatch", ...]
    },
    "diversity": {
      "by_type": {"article": 9, "monograph": 5, "textbook": 3, "npa": 2, "online": 3},
      "types_count": 5, "has_foreign": true, "pass": true
    },
    "gost_compliant": true
  }
}
```

**Когда реализовать**: Phase 2, при создании source_finder worker (SF-W1).
**Кто использует**: source_finder worker, academic-writer скилл (при RUNTIME-проверке), scorer.py (Stage 5 расширенный score_report).
**Дата**: Сессия 7
