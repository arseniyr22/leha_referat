## Задача: Создать скилл academic-writer

Это самый критичный скилл проекта. Он отвечает за КАЧЕСТВО СОДЕРЖАНИЯ академических текстов — глубину аргументации, предметную экспертизу, логические мосты, плотность фактов. Это НЕ антидетекция (для этого есть humanizer и pattern-auditor). Это про то, чтобы текст был содержательным, как у живого учёного.

### ВАЖНО: Это BUILD + RUNTIME скилл (решение D7)

**BUILD-режим** (при разработке):
- Используется при написании/улучшении Content Worker промптов
- Используется при доработке `prompts/academic_megaprompt.md`
- Проверяет: достаточно ли глубины, фактов, логических связей в генерируемом тексте

**RUNTIME-режим** (опционально):
- Может использоваться как content quality gate перед Stage 1
- Проверяет выходной текст Phase 0B на содержательную ценность

### Секция 1: Имя и триггер

```
Имя: academic-writer
Триггер: При написании/улучшении Content Worker промптов, при доработке megaprompt, при проверке качества содержания Phase 0B output
Команда: /academic-writer [domain] [section_type]
```

### Секция 2: Цель

Обеспечить максимальное качество СОДЕРЖАНИЯ академических текстов по 7 доменам. Каждый сгенерированный текст должен:
1. Создавать реальную интеллектуальную ценность (синтез, не компиляция)
2. Соответствовать доменным эпистемологическим нормам
3. Содержать конкретные факты, данные, имена, источники — не общие рассуждения
4. Иметь логические мосты между разделами (содержательные, не механические "Далее рассмотрим...")
5. Проходить проверку по 5-уровневой шкале глубины (БЛОК 7)

### Секция 3: Пошаговый процесс

**При BUILD-использовании (доработка промптов):**

```
Шаг 3.1: Определить domain из контекста
  → it_cs | law | psychology | economics | humanities | media | general

Шаг 3.2: Определить section_type
  → introduction | chapter_1 | chapter_2 | chapter_3 | conclusion | methodology | results | discussion | literature_review | full

Шаг 3.3: Загрузить доменный шаблон из reference/domain_templates.md
  → Для данного domain + section_type: обязательные элементы, минимальные метрики, типичные источники

Шаг 3.4: Загрузить чеклист плотности из reference/content_density_checklist.md
  → Минимумы на 500 слов: факты, цифры, имена, источники [N], логические мосты

Шаг 3.5: Загрузить шкалу глубины из reference/synthesis_depth_rubric.md
  → 5 уровней: (1) критическая оценка, (2) закономерности, (3) авторская модель, (4) рекомендации, (5) новый вопрос

Шаг 3.6: Проанализировать текущий промпт/текст на соответствие
  → Для каждого элемента чеклиста: ✅ есть / ❌ нет / ⚠️ слабо
  → Определить текущий уровень глубины (1-5)
  → Определить минимально необходимый уровень для данного stream_id

Шаг 3.7: Сгенерировать рекомендации
  → ЧТО добавить (конкретно, с примерами)
  → ГДЕ добавить (в какой секции промпта/текста)
  → ПОЧЕМУ (какое правило из megaprompt нарушено)

Шаг 3.8: Проверить на красные флаги
  → reference/quality_red_flags.md: БЛОК 7.6 + расширенные
  → Если хоть один красный флаг → HARD FAIL, конкретные инструкции по исправлению
```

**При RUNTIME-использовании (проверка Phase 0B output):**

```
Шаг R.1: Принять текст Phase 0B + params (stream_id, domain, language, level)
Шаг R.2: Прогнать по чеклисту content_density_checklist
Шаг R.3: Определить уровень глубины по synthesis_depth_rubric
Шаг R.4: Проверить на красные флаги
Шаг R.5: Вернуть отчёт: {score: 1-5, gaps: [...], red_flags: [...], recommendations: [...]}
```

### Секция 5: Правила

```
R1: НИКОГДА не переписывать текст — только анализировать и рекомендовать. Переписывание — работа Content Workers и HM workers.

R2: Фокус ТОЛЬКО на содержании. Стиль, антидетекция, паттерны P1-P53 — зона ответственности других скиллов (pattern-auditor, prompt-crafter). academic-writer проверяет: есть ли мысль, факты, логика, глубина.

R3: Каждая рекомендация ДОЛЖНА содержать конкретный пример. Не "добавьте больше фактов", а "в параграфе 2.3 утверждение 'санкции оказали влияние' заменить на 'по данным ЦБ РФ, прямые потери от санкций в 2022-2023 составили около 2.5% ВВП [3]'".

R4: Доменные правила имеют приоритет над общими. Если для it_cs нормально использовать пассивный залог ("модель была обучена"), не флагировать его — это доменная норма (P41).

R5: Минимальные уровни глубины по стримам ОБЯЗАТЕЛЬНЫ:
  - vkr: уровни 1-4 обязательны, 5 желателен
  - coursework: уровни 1-3 обязательны, 4 желателен
  - research: уровни 1-5 ВСЕ обязательны
  - abstract_paper: уровни 1-2 обязательны
  - text: уровни 1-2, стиль определяет глубину
  - essay: уровень 1, тезис = ценность
  - composition: уровень 1 минимум

R6: Красные флаги из БЛОКА 7.6 — HARD FAIL без исключений. Если параграф целиком состоит из пересказа одного источника — это провал содержания, неважно насколько хорош стиль.

R7: Логические мосты между разделами — содержательные, не механические. "Далее рассмотрим..." = красный флаг. Правильно: "Анализ в главе 1 показал пробел в [X]. Для его заполнения необходим эмпирический материал — именно его анализу посвящена следующая глава."

R8: Визуализации — не декорация. Каждая таблица/рисунок ОБЯЗАН иметь: (а) ссылку в тексте ДО появления, (б) абзац анализа ПОСЛЕ с конкретным выводом, (в) указание источника данных. Минимумы по стримам — из БЛОКА 16.2.

R9: При работе с русским текстом — учитывать все AC-правила (AC-1 through AC-15), особенно:
  - AC-2: GOST [N] формат цитирования, не author-year
  - AC-4: Пассивный залог допустим до 70% для русского академического
  - AC-5: Никаких метафор в vkr/coursework/research
```

### Секция 6: Self-Improvement

```
После каждого использования:
1. Записать: [ДАТА] [DOMAIN] [SECTION] [УРОВЕНЬ_ГЛУБИНЫ] [RED_FLAGS] [РЕКОМЕНДАЦИИ]
2. Если рекомендация была отвергнута — записать причину
3. Если рекомендация была принята — записать результат (улучшился ли текст)
4. Каждые 5 использований: проверить, не устарели ли доменные шаблоны (новые источники, новые стандарты)
5. Если появился новый домен не покрытый шаблонами → создать шаблон по аналогии с ближайшим
```

## Шаг 4 — Создать reference/domain_templates.md

Для каждого из 7 доменов создай ПОЛНЫЙ шаблон со следующей структурой:

```markdown
# Доменные шаблоны содержания

## IT / Computer Science (`it_cs`)

### Обязательные элементы по секциям

**Введение:**
- Конкретная техническая/прикладная задача, не решённая существующими инструментами
- Стек технологий с обоснованием выбора (не просто "использовался Python", а "Python выбран из-за экосистемы ML-библиотек (scikit-learn, PyTorch) и зрелости NLP-инструментов (spaCy, NLTK)")
- Метрики успеха определены заранее (RMSE, F1-score, latency, throughput)

**Глава 1 (теоретическая):**
- Обзор существующих решений: название → автор → год → метрика → ограничение
- Сравнительная таблица подходов (минимум 3-5 подходов, минимум 4 параметра сравнения)
- Архитектурные паттерны, применимые к задаче

**Глава 2 (практическая):**
- Описание архитектуры: компоненты, взаимодействие, потоки данных
- UML/ER-диаграммы (ссылки-плейсхолдеры)
- Результаты: ВСЕГДА с числовыми метриками + сравнение с baseline
- Формат результатов: "[Метод] достиг [метрика] = [значение] (baseline: [значение], улучшение: [%])"

**Глава 3 (рекомендательная):**
- Конкретные рекомендации по масштабированию/оптимизации
- Ожидаемый эффект с метриками

**Минимальная плотность (на 500 слов):**
- Названия алгоритмов/фреймворков: ≥ 3
- Числовые метрики: ≥ 2
- Ссылки [N]: ≥ 3
- Сравнения с baseline/альтернативой: ≥ 1

**Типичные источники:** IEEE/ACM papers, GitHub repos (cited by version), API documentation, стандарты ISO/ГОСТ

---

## Юриспруденция (`law`)

### Обязательные элементы по секциям

**Введение:**
- Конкретное противоречие/пробел в законодательстве или правоприменении
- Нормативно-правовая база: перечень ключевых НПА с номерами статей
- Объект — сфера правоотношений; предмет — конкретный аспект регулирования

**Глава 1 (теоретическая):**
- Эволюция правового института: хронология ключевых изменений
- Доктринальные подходы: минимум 3 правовые школы/позиции с именами учёных
- Сравнительно-правовой анализ (РФ vs 2-3 другие юрисдикции)

**Глава 2 (аналитическая):**
- Действующее законодательство: ПОСТАТЕЙНЫЙ анализ (не общий обзор)
- Проблемы правоприменения с конкретными примерами
- Судебная практика: минимум 5 решений с реквизитами (суд, дата, номер дела)

**Глава 3 (рекомендательная):**
- Конкретные предложения по совершенствованию: какую статью/пункт изменить, как
- Обоснование через выявленные в Главе 2 проблемы

**Минимальная плотность (на 500 слов):**
- Ссылки на конкретные статьи законов: ≥ 4
- Имена учёных-правоведов: ≥ 2
- Реквизиты судебных решений: ≥ 1 (в главе 2 — ≥ 3)
- Ссылки [N]: ≥ 5

**Типичные источники:** кодексы, ФЗ, постановления, определения КС/ВС РФ, научные статьи, комментарии к законодательству

---

## Психология / Педагогика (`psychology`)

### Обязательные элементы по секциям

**Введение (три вектора актуальности):**
1. Социальный контекст: конкретная проблема в обществе
2. Теоретический пробел: что не изучено
3. Практическая потребность: какой инструмент/методика отсутствует

**Глава 1 (теоретическая):**
- Обзор теорий: минимум 3 концептуальных подхода (бихевиоризм, когнитивный, гуманистический и т.д.)
- Систематизация: в чём подходы сходятся, в чём расходятся
- Операционализация понятий: определения через измеримые индикаторы

**Глава 2 (эмпирическая — если empirical):**
- Выборка: N, возраст (M ± SD), пол (%), критерии включения/исключения
- Инструменты: полные названия методик + авторы + год + шкалы
- Результаты: ТОЛЬКО факты, данные, таблицы — БЕЗ интерпретации
- Статистика: названия тестов + значения + уровень значимости (p < 0.05)
- Таблицы результатов с доверительными интервалами

**Обсуждение (отдельно от результатов!):**
- Интерпретация результатов через теории из Главы 1
- Сопоставление с данными других исследований
- Ограничения: угрозы валидности, проблемы выборки

**Минимальная плотность (на 500 слов):**
- Имена исследователей с годами: ≥ 3
- Названия методик/тестов: ≥ 1
- Статистические показатели: ≥ 2 (в главе результатов — ≥ 5)
- Ссылки [N]: ≥ 4

**Типичные источники:** peer-reviewed journals (РИНЦ, Scopus, WoS), диссертации, учебники по методологии

---

## Экономика / Менеджмент (`economics`)

### Обязательные элементы по секциям

**Введение:**
- Конкретные статистические данные, рыночная ситуация, кризис
- Объект — организация/отрасль/регион; предмет — конкретный экономический механизм

**Глава 1 (теоретическая):**
- Обзор экономических теорий/моделей: не менее 3 подходов
- Определения ключевых понятий через авторитетные источники
- Зарубежный опыт: минимум 2 страны/компании

**Глава 2 (аналитическая):**
- Финансовый анализ: таблицы показателей, динамика, тренды
- Данные Росстата, ЦБ РФ, отраслевых ассоциаций — ОБЯЗАТЕЛЬНО с номерами [N]
- Авторские расчёты: метод расчёта + исходные данные + результат
- Графики динамики показателей (минимум 2-3)

**Глава 3 (рекомендательная):**
- Конкретные мероприятия: что → кому → каким инструментом → с каким ожидаемым результатом
- Экономическое обоснование: расчёт затрат и ожидаемого эффекта
- НЕ "необходимо совершенствовать", а "внедрение CRM-системы (стоимость ~500 тыс. руб.) позволит сократить время обработки заказов на 35%, что при текущем объёме заказов даст экономию ~1.2 млн руб./год"

**Минимальная плотность (на 500 слов):**
- Числовые показатели (%, руб., тонны, и т.д.): ≥ 4
- Названия организаций/источников данных: ≥ 2
- Ссылки [N]: ≥ 4
- Сравнения периодов/объектов: ≥ 1

**Типичные источники:** Росстат, ЦБ РФ, Минфин, World Bank, IMF, отраслевые отчёты, финансовая отчётность

## Гуманитарные науки (`humanities`)

### Обязательные элементы по секциям

**Введение — через историографию:**
- Кто и когда изучал тему
- Какие периоды/аспекты остались без внимания
- Первичные источники, на которых строится работа

**Глава 1:**
- Историографический обзор: школы/традиции → представители → ключевые тезисы
- Критический анализ: не "учёный N считает X", а оценка позиции N
- Периодизация (если историческая работа)

**Глава 2:**
- Анализ первичных источников (архивы, оригинальные тексты, документы)
- Сопоставление позиций разных авторов по конкретным вопросам
- Авторская интерпретация: собственный вклад в понимание темы

**Минимальная плотность (на 500 слов):**
- Первичные источники: ≥ 1
- Имена учёных/исторических деятелей: ≥ 3
- Даты/периоды: ≥ 2
- Ссылки [N]: ≥ 4

**Типичные источники:** архивные документы, мемуары, монографии, академические журналы, диссертации

## Медиа (`media`)

### Обязательные элементы по секциям

**Введение:**
- Трансформация медиарынка: конкретные данные о медиапотреблении
- Исследовательский вопрос или гипотеза — ОБЯЗАТЕЛЬНЫ
- Научная новизна — ОБЯЗАТЕЛЬНА

**Глава 1 (теоретическая):**
- Медиатеории: минимум 3 подхода (повестка дня, фрейминг, теория культивации и т.д.)
- Определения через медиаисследователей (McLuhan, Castells, Jenkins и т.д.)

**Глава 2 (эмпирическая):**
- Контент-анализ: кодировочная таблица, единицы анализа, выборка контента
- Аудиторные метрики: охват, вовлечённость, конверсия
- Таблицы + графики результатов

**Минимальная плотность (на 500 слов):**
- Медиаплатформы/издания: ≥ 2
- Аудиторные метрики: ≥ 2
- Имена медиаисследователей: ≥ 2
- Ссылки [N]: ≥ 3

**Типичные источники:** Mediascope, ВЦИОМ, Reuters Institute, Pew Research, академические журналы по медиа

## General (`general`)

Применяется когда домен не определён однозначно. Берутся минимальные требования:

**Минимальная плотность (на 500 слов):**
- Конкретные факты/данные: ≥ 3
- Имена/организации: ≥ 2
- Ссылки [N]: ≥ 3
- Логические мосты между параграфами: ≥ 1
```

**КРИТИЧЕСКИ ВАЖНО**: Каждый доменный шаблон должен содержать КОНКРЕТНЫЕ примеры формулировок. Не абстрактные "добавьте факты", а "формулировка X заменяется на формулировку Y с конкретными данными".

## Шаг 5 — Создать reference/synthesis_depth_rubric.md

Извлечь ВЕРБАТИМ из `prompts/academic_megaprompt.md` БЛОК 7 (секции 7.1-7.6) и дополнить:

```markdown
# Шкала глубины синтеза (5 уровней)

## Источник: БЛОК 7 academic_megaprompt.md

[ЗДЕСЬ ВЕРБАТИМ КОПИЯ БЛОКА 7.1-7.6]

## Дополнение: как определить уровень

### Экспресс-тест (для быстрой оценки параграфа):

**Уровень 0 (ПРОВАЛ):** Параграф можно было написать, прочитав только одну статью из Википедии. Нет авторской позиции, нет сопоставления, нет вывода.

**Уровень 1 (минимум):** Параграф содержит оценку хотя бы одной позиции: "Подход Иванова [1] продуктивен в части..., однако ограничен тем, что..."

**Уровень 2:** Параграф содержит паттерн, которого не было ни в одном из источников по отдельности: "Анализ кейсов [4, 7, 12] позволяет выявить общую закономерность: ..."

**Уровень 3:** Параграф/секция предлагает авторскую типологию/модель/классификацию, которой не было в литературе.

**Уровень 4:** Рекомендации конкретны, адресны и ВЫТЕКАЮТ из анализа: "что → кому → каким инструментом → ожидаемый результат".

**Уровень 5:** Работа формулирует НОВЫЙ вопрос, ставший видимым благодаря проведённому анализу.

### Минимальные уровни по стримам (таблица из БЛОКА 7.3 — ВЕРБАТИМ)

[ЗДЕСЬ ВЕРБАТИМ ТАБЛИЦА 7.3]
```

## Шаг 6 — Создать reference/content_density_checklist.md

```markdown
# Чеклист плотности содержания

## Минимальные метрики на 500 слов (общие)

| Метрика | Минимум | Что считать |
|---------|---------|------------|
| Конкретные факты | ≥ 3 | Датированные события, статистика, названия, числа |
| Ссылки [N] | ≥ 3 | Каждая ссылка — конкретный источник, не "по мнению экспертов" |
| Имена (учёные, организации) | ≥ 2 | Полные имена или устоявшиеся сокращения с расшифровкой |
| Числовые данные | ≥ 2 | Проценты, суммы, даты, объёмы — не "значительный", а "35%" |
| Логические мосты | ≥ 1 | Содержательная связь с предыдущим/следующим разделом |

## Доменные надбавки

(Добавить к общим минимумам)

| Домен | Дополнительные метрики |
|-------|----------------------|
| it_cs | Названия алгоритмов/фреймворков ≥ 3; числовые метрики ≥ 2; сравнения с baseline ≥ 1 |
| law | Статьи законов ≥ 4; реквизиты судебных решений ≥ 1; имена правоведов ≥ 2 |
| psychology | Названия методик ≥ 1; статистические показатели ≥ 2; имена исследователей ≥ 3 |
| economics | Числовые показатели ≥ 4; источники данных (Росстат, ЦБ) ≥ 2; сравнения периодов ≥ 1 |
| humanities | Первичные источники ≥ 1; даты/периоды ≥ 2; имена учёных/деятелей ≥ 3 |
| media | Медиаплатформы ≥ 2; аудиторные метрики ≥ 2; имена медиаисследователей ≥ 2 |

## Антиметрики (красные флаги плотности)

| Антиметрика | Описание | Порог |
|-------------|----------|-------|
| Вакантные ссылки | "Эксперты считают...", "Ряд учёных полагает..." без [N] | 0 допустимо |
| Абстрактные утверждения | "Оказал значительное влияние" без цифр | ≤ 1 на 500 слов |
| Пустые оценки | "Является актуальной проблемой" | 0 допустимо |
| Пересказ без оценки | Параграф = изложение одного источника | 0 допустимо |
```

Извлечь ВЕРБАТИМ БЛОК 7.6 из `prompts/academic_megaprompt.md` + БЛОК 12 (запрещённые паттерны) и ДОПОЛНИТЬ:

```markdown

[ВЕРБАТИМ КОПИЯ БЛОКА 12 — структурные + логические]

## Расширенные красные флаги (дополнение)

### Содержательные

| Красный флаг | Описание | Как исправить |
|-------------|----------|---------------|
| Пустая актуальность | "В наше время...", "В условиях современности..." | Конкретный факт + дата + источник |
| Пустые задачи | Глаголы "изучить", "рассмотреть", "описать" | Заменить на: выявить, определить, оценить, разработать |
| Зеркальное заключение | Заключение = перефраз введения | Конкретные НОВЫЕ выводы из каждой главы |
| Рекомендации без обоснования | "Необходимо улучшить..." | ЧТО → КОМУ → ЧЕМ → КАКОЙ РЕЗУЛЬТАТ |
| Висячая визуализация | Таблица/рисунок без анализа после | Абзац с конкретным выводом |
| Компиляция вместо синтеза | "A считает X. B считает Y. Тема изучается." | Сопоставить A и B: в чём расхождение? Кто убедительнее? |
| Пересказ без источника | Утверждение без [N] — неверифицируемо | Добавить [N] или удалить |
| Абстрактный прогноз | "Тема требует дальнейшего изучения" | Конкретный вопрос + метод + ожидание |

### Доменные красные флаги

**it_cs:** Результаты без метрик; нет сравнения с baseline; код в теле работы (только в приложениях)
**law:** Моральные суждения без правовой основы; ссылки без номеров статей; нет судебной практики в главе 2
**psychology:** Результаты и обсуждение смешаны в одном разделе; нет описания выборки; нет названий методик
**economics:** Рекомендации без экономического обоснования; нет данных Росстата/ЦБ; нет динамики показателей
**humanities:** Хронологический пересказ без аналитического стержня; нет первичных источников; нет историографического обзора
**media:** Нет исследовательского вопроса; нет аудиторных метрик; нет контент-анализа
```
