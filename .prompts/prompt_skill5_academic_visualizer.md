### Секция 2: Цель

Обеспечить что каждая визуализация (таблица, рисунок, схема) в генерируемом тексте:
1. Решает конкретную аналитическую задачу (не декорация)
2. Соответствует стриму (минимумы: vkr 5-15, coursework 3-6, research 3-8, abstract_paper 1-3)
3. Подписана по ГОСТ (таблицы — НАД, рисунки — ПОД, без точки в конце)
4. Имеет указание источника данных БЕЗ ИСКЛЮЧЕНИЙ
5. Интегрирована в текст (ссылка ДО + анализ ПОСЛЕ)
6. Соответствует доменному профилю (it_cs=UML/ER, law=таблицы норм, psychology=графики с CI, economics=динамика+SWOT, humanities=хронология, media=аудиторные метрики)
7. Соответствует техническим требованиям (SVG/300dpi, читаемый шрифт ≥9pt, ч/б-совместимость, без 3D-эффектов)
8. **МОЖЕТ БЫТЬ СОЗДАНА ПРОГРАММНО** когда данные доступны — через matplotlib, graphviz, mermaid, или python-docx

Что НЕ входит в зону ответственности:
- Качество ТЕКСТА между визуализациями — зона academic-writer
- Антидетекция — зона pattern-auditor / humanizer
- Построение промптов — зона prompt-crafter

### Секция 3: Пошаговый процесс

#### BUILD-режим

**Шаг 3.1: Определить параметры**
```
stream_id: vkr | coursework | research | abstract_paper | text | essay | composition
domain: it_cs | law | psychology | economics | humanities | media | general
language: ru | en (default: ru)
level: bachelor | master (default: bachelor)
```
Если параметры не указаны явно — определить по контексту темы. При невозможности — запросить.

**Шаг 3.2: Загрузить минимумы по стриму**
Из `reference/stream_minimums.md`:
- vkr bachelor: 5–10 таблиц/рисунков
- vkr master: 8–15
- coursework: 3–6
- research: 3–8 (зависит от типа)
- abstract_paper: 1–3 (если есть данные)
- text: по контексту (если данные/статистика/сравнение — включить)
- essay: не требуется (только если усиливает аргумент)
- composition: нет

**Шаг 3.3: Загрузить доменный профиль визуализаций**
Из `reference/domain_viz_profiles.md`:
- Какие типы визуализаций характерны для данного домена
- Какие ОБЯЗАТЕЛЬНЫ (для данного домена + stream)
- Какие запрещены или нетипичны
- Конкретные примеры хороших визуализаций для этого домена

**Шаг 3.4: Проверить 10 триггеров автоматического создания визуализации**
Из `reference/visualization_triggers.md`:
1. Сравнение 3+ объектов по нескольким параметрам → ТАБЛИЦА
2. Динамика показателя во времени → ЛИНЕЙНЫЙ ГРАФИК
3. Доли / структура / распределение → КРУГОВАЯ или СТОЛБЧАТАЯ ДИАГРАММА
4. Причинно-следственные связи → БЛОК-СХЕМА или СХЕМА СВЯЗЕЙ
5. Процесс / алгоритм / последовательность → БЛОК-СХЕМА
6. Классификация / типология → ТАБЛИЦА или ДРЕВОВИДНАЯ СХЕМА
7. Числовые результаты эксперимента → ТАБЛИЦА + ГРАФИК
8. Сравнение «до и после» → СРАВНИТЕЛЬНАЯ ТАБЛИЦА
9. Географическое распределение → КАРТА или ТАБЛИЦА ПО РЕГИОНАМ
10. Корреляция двух переменных → SCATTER PLOT

Для каждого триггера, встреченного в тексте: проверить, создана ли соответствующая визуализация. Если нет — рекомендовать И предложить конкретную реализацию (шаблон кода или markdown-таблицу).

**Шаг 3.5: Проверить качество каждой визуализации по чеклисту**
Из `reference/visualization_checklist.md` (9 пунктов):
- [ ] Решает конкретную задачу (не «для красоты»)
- [ ] Тип адекватен данным (не круговая для 15 сегментов; не линейный для категориальных)
- [ ] Подпись по стандарту: «Таблица N — Название» НАД / «Рисунок N — Название» ПОД (без точки)
- [ ] Источник данных указан
- [ ] В тексте ДО — ссылка на визуализацию
- [ ] ПОСЛЕ — абзац анализа (конкретный вывод, НЕ «как видно из таблицы»)
- [ ] Текст на графике читаем, ничто не накладывается
- [ ] Работает в ч/б печати
- [ ] Данные актуальны (или оговорка о периоде)

**Шаг 3.6: Проверить подписание по языковому гейту**
Из `reference/caption_format.md`:
- `language=ru`: «Таблица N — Название» / «Рисунок N — Название» / «Источник: составлено автором на основе [N]»
- `language=en`: «Table N — Title» / «Figure N — Title» / «Source: compiled by the author based on [N]»
- Нумерация сквозная по всему тексту (таблицы отдельно, рисунки отдельно)
- Без точки в конце подписи

**Шаг 3.7: Проверить размещение по главам**
- Глава 1 (теоретическая): таблицы сравнения подходов, схемы моделей, хронологические таблицы
- Глава 2 (аналитическая): ОСНОВНАЯ масса — данные, динамика, результаты
- Глава 3 (рекомендательная): схемы решений, таблицы ожидаемых результатов, дорожные карты
- Приложения: громоздкие таблицы (>1 стр.), вспомогательные графики

**Шаг 3.8: Проверить технические требования**
Из `reference/technical_requirements.md`:
- Формат: SVG или ≥300 dpi
- Шрифт: Times New Roman или Arial, ≥9pt
- Цвета: академическая палитра (синий, тёмно-серый, бордо, тёмно-зелёный)
- Фон: белый, без градиентов
- Без 3D-эффектов
- Таблицы: не более 7–8 колонок
- Круговая: не более 6–7 сегментов
- Линейный: не более 3 линий
- Scatter: подписанные оси с единицами измерения
- Легенда не перекрывает данные

**Шаг 3.9: Сгенерировать рекомендации**
Формат:
```
[ТАБЛИЦА/РИСУНОК/СХЕМА] [N]: [Статус: ✅ OK / ❌ FAIL / ⚠️ WARN]
  Проблема: [конкретно]
  Решение: [конкретно — что изменить, добавить, удалить]
  Правило: [ссылка на БЛОК 16.X]

ОБЩИЙ ИТОГ:
  Визуализаций: [текущее] / [минимум] → [PASS/FAIL]
  Пропущенные триггеры: [список — какие контексты требуют визуализации, но её нет]
```

**Это КРИТИЧЕСКИ ВАЖНАЯ часть скилла.** CREATE-режим отвечает за программную генерацию визуализаций наивысшего качества.

**Шаг C.1: Определить тип визуализации из контекста**
По триггеру (из шага 3.4) или по прямому запросу определить:

- Для какого домена и стрима

**Шаг C.2: Выбрать технологию создания**

Решающее дерево:

```
Тип визуализации → Технология:

ТАБЛИЦЫ (любые):
  → При генерации текста: markdown формат (| ... |)
  → При рендере в .docx: python-docx (Table class)

ЛИНЕЙНЫЕ ГРАФИКИ / СТОЛБЧАТЫЕ / КРУГОВЫЕ / SCATTER / ГИСТОГРАММЫ:
  → matplotlib + matplotlib.font_manager (для Times New Roman)
  → Выход: SVG (предпочтительно) или PNG 300dpi
  → Обязательные настройки:
    - plt.rcParams['font.family'] = 'serif'
    - plt.rcParams['font.serif'] = ['Times New Roman']
    - plt.rcParams['font.size'] = 10
    - plt.rcParams['figure.dpi'] = 300
    - plt.rcParams['axes.grid'] = True
    - plt.rcParams['grid.alpha'] = 0.3
    - plt.style.use('seaborn-v0_8-whitegrid') или аналог
    - Палитра: ['#2C5F8A', '#4A4A4A', '#8B1A1A', '#2D6A2D', '#D4A017']

БЛОК-СХЕМЫ / АЛГОРИТМЫ / ОРГАНИЗАЦИОННЫЕ СТРУКТУРЫ:
  → graphviz (через python-graphviz)
  → Альтернатива: mermaid (если graphviz недоступен)
  → Для it_cs: ГОСТ 19.701-90 символы (прямоугольник=процесс, ромб=решение, параллелограмм=ввод/вывод)
  → Выход: SVG или PNG 300dpi

UML-ДИАГРАММЫ (для it_cs):
  → graphviz или PlantUML (если доступен)
  → Типы: class diagram, sequence diagram, use case, activity
  → Шрифт: Arial или Courier для кода

ER-ДИАГРАММЫ (для it_cs):
  → graphviz (записи + связи)
  → Обозначения кардинальности: 1, 0..1, 1..*, 0..*

SWOT-МАТРИЦЫ (для economics/management):
  → matplotlib (4 квадранта через GridSpec или subplot)
  → Или python-docx (таблица 2x2 с заливкой)

ХРОНОЛОГИЧЕСКИЕ ТАБЛИЦЫ:
  → markdown (для текста) или python-docx Table (для .docx)
  → Колонки: Год/Период | Событие | Значение/Результат

MIND MAP / СХЕМЫ СВЯЗЕЙ:
  → graphviz (mode: neato или fdp для нелинейных)
```

#### COMPLEX-режим (комплексные визуализации)

Для случаев, когда один тип визуализации недостаточен:

**Шаг X.1: Определить тип комплексной визуализации**
- Комбинированный график (столбцы + линия) — например: объём продаж (столбцы) + темп роста (линия)
- Таблица + мини-спарклайны — для больших массивов данных
- Мультипанельный рисунок (subplots) — для сравнения нескольких переменных
- Каскадная диаграмма (waterfall) — для факторного анализа
- Тепловая карта (heatmap) — для матриц корреляций
- Диаграмма Ганта — для дорожных карт и планов
- Sankey-диаграмма — для потоков (финансовых, информационных)

**Шаг X.2: Проверить обоснованность сложности**
Правило: комплексная визуализация оправдана ТОЛЬКО если простая не передаёт нужную связь. Если можно разбить на 2 простых — разбить.

**Шаг X.3: Применить доменные ограничения**
- it_cs: допустимы все комплексные типы
- law: только таблицы + блок-схемы (без тепловых карт, sankey)
- psychology: мультипанельные + heatmap корреляций + forest plot
- economics: все комплексные типы, waterfall ОСОБЕННО приветствуется
- humanities: только хронологические + mind map (без сложных графиков)
- media: допустимы комбинированные и мультипанельные

**Шаг X.4: Создать с проверкой читаемости**
После создания: проверить что при масштабе 100% все подписи читаемы, ничто не накладывается, легенда не перекрывает данные. Если нет — увеличить figsize или перенести легенду.

#### ADAPTIVE-режим (гибкость при нетипичных сценариях)

**Шаг A.1: Междоменные работы**
Если тема на стыке доменов (например, "Цифровизация правовых процессов" = law + it_cs):
- Взять обязательные визуализации ИЗ ОБОИХ доменов
- Минимум = MAX(domain1_min, domain2_min)
- Типы = UNION(domain1_types, domain2_types)
- Антипаттерны = UNION(domain1_anti, domain2_anti)

**Шаг A.2: Нестандартные стримы**
Если stream_id = "text" или "essay" с визуально-насыщенной темой (напр. "Инфографика в медиа"):
- Повысить минимум до уровня coursework (3–6)
- Добавить визуализации как примеры внутри текста

**Шаг A.3: Недоступность данных**
Если для триггера нет конкретных данных (нет числовых значений в тексте):
- Для таблиц: создать структуру с плейсхолдерами [?] и пометкой "Данные уточняются"
- Для графиков: создать плейсхолдер `[РИСУНОК N — Название (данные из источника [M])]`
- НИКОГДА не выдумывать данные — лучше плейсхолдер чем фальшивые числа

**Шаг A.4: Масштабирование для больших работ**
Для работ >50 страниц (магистерские, диссертации):
- Распределение: ~1 визуализация на 4–6 страниц текста
- В аналитической главе: ~1 визуализация на 2–3 страницы
- В теоретической главе: ~1 визуализация на 5–8 страниц
- Баланс типов: не более 60% одного типа (не всё таблицы)

#### 4.3 `reference/caption_format.md`
Полные правила подписей по ГОСТ + EN:

**Таблицы:**
- RU: «Таблица N — Название» (НАД таблицей, без точки в конце)
- EN: «Table N — Title» (above table, no period)
- Нумерация сквозная по всему документу
- Источник: ПОСЛЕ таблицы, отдельная строка: «Источник: [N]» или «Составлено автором на основе [N1, N2]»

**Рисунки (графики, диаграммы, схемы):**
- RU: «Рисунок N — Название» (ПОД рисунком, без точки в конце)
- EN: «Figure N — Title» (below figure, no period)
- Нумерация сквозная, отдельно от таблиц
- Источник: под подписью: «Источник: [N]» или «Составлено автором»

**Языковой гейт (вербатим из БЛОК 16.8):**
| Русский | Английский |
|---|---|
| «Таблица N — Название» | «Table N — Title» |
| «Рисунок N — Название» | «Figure N — Title» |
| «Источник: составлено автором» | «Source: compiled by the author» |
| «Источник: составлено автором на основе [N]» | «Source: compiled by the author based on [N]» |
| «Источник: [N]» | «Source: [N]» |

**Правило интеграции (вербатим из БЛОК 16.5):**
```
[Абзац анализа, подводящий к визуализации]
→ Ссылка: «Динамика показателя X представлена на рисунке 3.»
→ [Рисунок 3 — Название]
→ [Абзац с анализом: выводы, закономерности, аномалии — НЕ повтор данных]
```

**ЗАПРЕЩЕНО (вербатим из БЛОК 16.5):**
- Визуализация без предварительной ссылки в тексте
- Визуализация без последующего анализа
- «Как видно из таблицы 2...» без конкретного вывода — это ПУСТАЯ ФРАЗА
- Дублировать в тексте то, что видно из таблицы/графика (вместо этого — ИНТЕРПРЕТАЦИЯ)

#### 4.4 `reference/technical_requirements.md`
Технические требования к визуализациям (вербатим из БЛОК 16.7 + дополнения):

**Общие:**
- Формат: SVG (предпочтительно) или ≥300 dpi PNG
- Шрифт: Times New Roman или Arial, не менее 9pt на графике
- Цветовая палитра: сдержанная, академическая
  - Основная: ['#2C5F8A', '#4A4A4A', '#8B1A1A', '#2D6A2D', '#D4A017']
  - Расширенная (до 8 серий): + ['#6B4C9A', '#C0392B', '#1ABC9C']
  - Ч/Б альтернатива: linestyles ['-', '--', '-.', ':'] + markers ['o', 's', '^', 'D'] + hatches ['///', '...', 'xxx', '\\\\\\']
- Фон: белый, без градиентов
- Рамка: тонкая серая линия или без рамки
- Соотношение сторон: ~4:3 или ~16:9 для графиков; таблицы — по содержимому
- Текст НЕ накладывается — проверять при любом масштабе
- Легенда НЕ перекрывает данные
- Подписи осей полностью видны
- **ОБЯЗАТЕЛЬНО**: работает в ч/б печати (паттерны/штриховка вместо цветов)
- **НИКАКИХ 3D-эффектов** (научный стандарт)
- Сетка: лёгкая, ненавязчивая (светло-серая, alpha=0.3)

**Таблицы:**
- Не более 7–8 колонок (иначе разбить)
- Числовые данные выровнены по правому краю или десятичной точке
- Каждая колонка имеет чёткий заголовок с единицами измерения
- Границы: все ячейки с границами, шапка — жирная нижняя линия
- Шрифт в таблице = основной шрифт документа (Times New Roman 14pt для ГОСТ, или 12pt если данных много)

**Графики:**
- Линейный: не более 3 линий (если больше — разбить на 2 графика или мультипанель)
- Круговая: не более 6–7 сегментов (если больше — объединить мелкие в "Прочее")
- Столбчатая: подписи значений на/над столбцами
- Оси подписаны: название переменной + единица измерения
- Тики осей: не более 10 на оси (если больше — увеличить шаг)
- Для временных рядов: формат дат единообразный (2019, 2020, 2021 или Q1'19, Q2'19...)

**Схемы:**
- Единый стиль оформления на протяжении всей работы
- Стрелки с подписями (что означает связь)
- Блоки с чёткими краткими подписями
- Не более 10–12 элементов на одной схеме
- Направление: сверху вниз или слева направо (не смешивать)

### Секция 5: Правила

**R1: Каждая визуализация = инструмент аргументации, не украшение.** Если визуализацию можно убрать без потери для аргумента — её не должно быть. Если данные понятнее словами — не создавать визуализацию.

**R2: Подпись СТРОГО по ГОСТ** (AC-2 для визуализаций):
- Таблицы: «Таблица N — Название» (НАД, без точки) — RU
- Рисунки: «Рисунок N — Название» (ПОД, без точки) — RU
- Язык подписей = язык работы (AC-8 адаптация)

**R3: Правило «ссылка → визуализация → анализ» — БЕЗ ИСКЛЮЧЕНИЙ.**
- ДО визуализации: ссылка в тексте, подводящая к ней
- ПОСЛЕ визуализации: абзац с конкретным выводом (НЕ «как видно из таблицы»)
- Если визуализация без анализа → HARD FAIL

**R4: Источник данных — БЕЗ ИСКЛЮЧЕНИЙ.** Каждая визуализация обязана иметь «Источник: ...». Варианты:
- «Источник: [N]» — из конкретного цитируемого источника
- «Составлено автором на основе [N1, N2]» — авторская обработка
- «Источник: данные Росстата» — открытые данные

**R5: Минимумы — жёсткие для обязательных стримов.** Если визуализация «обязательна» (vkr, coursework, research) и count < minimum → FAIL. Для «желательных» (abstract_paper) и «по контексту» (text) — WARN.

**R6: Доменный профиль определяет типы.** it_cs БЕЗ UML/блок-схемы в ВКР → WARN. economics БЕЗ графика динамики → WARN. psychology БЕЗ графика результатов с CI → WARN.

**R7: НЕ дублировать текст в визуализации.** Если таблица содержит данные — в тексте после НЕ пересказывать числа. Вместо этого — ИНТЕРПРЕТИРОВАТЬ: закономерности, аномалии, выводы.

**R8: formatter.py gap.** На данный момент formatter.py НЕ умеет парсить markdown-таблицы и плейсхолдеры рисунков. При BUILD-режиме доработки formatter.py:
- Markdown `| ... |` → ГОСТ Word-таблица с подписью НАД
- `[РИСУНОК N — Название]` → Word image block с подписью ПОД
- Нумерация: таблицы и рисунки нумеруются сквозной, раздельно
- `Источник: ...` → мелкий курсив под таблицей/рисунком

**R9: Приоритет источников данных** (БЛОК 16.4):
1. Авторские расчёты → «Составлено автором»
2. Открытые данные (Росстат, ЦБ, Евростат, World Bank) → ссылка
3. Данные из источников → «Источник: [N]»
4. Данные организации → «По данным [организация]»

**R10: НИКОГДА не выдумывать данные.** Если числовые данные недоступны — создать плейсхолдер с пометкой `[данные из источника [N]]`. Лучше плейсхолдер чем фальшь.

**R11: Качество генерации = качество ручной работы.** Каждый matplotlib-график, graphviz-диаграмма и python-docx-таблица должны выглядеть как если бы их создал профессиональный исследователь вручную. Никаких default стилей, никаких rainbow палитр, никаких необрезанных подписей.

**R12: Баланс типов.** В работе не должно быть >60% визуализаций одного типа. Если 5 из 6 — таблицы → добавить графики или схемы.

**R13: Комплексные визуализации — только при обоснованной необходимости.** Если простая визуализация достаточна — не усложнять. Комплексные типы (dual axis, waterfall, heatmap) допустимы только когда они раскрывают связь, невидимую в простых типах.

## Пояснения для тебя, Артём:

### Что изменилось по сравнению с первой версией

Провёл критический аудит и нашёл **5 серьёзных gap'ов** в первой версии. Все исправлены:

#### GAP 1: Скилл только КОНТРОЛИРОВАЛ, но не СОЗДАВАЛ визуализации
**Проблема:** Первая версия была «инспектором» — проверяла чеклисты, считала минимумы, но не давала Claude Code инструкций КАК создавать графики, диаграммы, таблицы.
**Исправление:** Добавлен полноценный **CREATE-режим** (шаги C.1–C.5) с решающим деревом выбора технологии, ГОСТ-совместимыми настройками matplotlib, шаблонами кода, и автоматической валидацией после создания.

#### GAP 2: Нет рецептов реализации
**Проблема:** Даже если Claude Code знает что нужен «линейный график», без конкретного кода он создаст график с default палитрой (rainbow), sans-serif шрифтом и 72 dpi. Это провал качества.
**Исправление:** Добавлен новый файл **`reference/implementation_recipes.md`** с 10 готовыми рецептами — полный рабочий код для каждого типа визуализации (линейный, столбчатый, круговой, scatter, блок-схема ГОСТ, ER, SWOT, combo, python-docx таблица, хронология).

#### GAP 3: Нет комплексных визуализаций
**Проблема:** Первая версия знала только про простые типы (таблица, линейный, круговая). Но комплексные визуализации (waterfall, heatmap, dual-axis, sankey, mermaid) — это то, что отличает сильную работу от средней.
**Исправление:** Добавлен **COMPLEX-режим** (шаги X.1–X.4) с каталогом комплексных типов, проверкой обоснованности сложности, и доменными ограничениями (law = без heatmap, humanities = без сложных графиков).

#### GAP 4: Нет гибкости для нетипичных случаев
**Проблема:** Жёсткая привязка к 7 доменам. А что если тема на стыке (law + it_cs)? Или работа на 100 страниц? Или пользователь хочет нестандартный стиль?
**Исправление:** Добавлен **ADAPTIVE-режим** (шаги A.1–A.5) — междоменные работы (UNION типов), масштабирование для больших работ (формула ~1 визуализация на 4-6 стр.), обработка недоступных данных (плейсхолдеры вместо фальшивых цифр), пользовательские предпочтения (WARN вместо FAIL).

#### GAP 5: Нет правил для программной генерации
**Проблема:** Чеклист из БЛОК 16.9 проверяет «читаем ли текст на графике», но не проверяет «правильный ли шрифт в matplotlib», «300 dpi или 72 dpi», «academic palette или rainbow».
**Исправление:** В чеклист добавлена секция **«Чеклист качества генерации»** (7 пунктов, №16-22) — matplotlib шрифт, dpi, сетка, палитра, graphviz параметры, python-docx границы.

### Итого: что теперь умеет скилл

| Режим | Что делает | Когда |
|-------|-----------|-------|
| BUILD | Подсказывает правила при написании промптов/кода | Phase 2-3 разработки |
| RUNTIME | Проверяет готовый текст на количество и качество виз. | Phase 0B output |
| CREATE | Генерирует визуализации с кодом (matplotlib, graphviz, python-docx) | Когда данные доступны |
| COMPLEX | Создаёт комплексные визуализации (waterfall, heatmap, dual-axis) | Когда простых недостаточно |
| ADAPTIVE | Адаптируется под нетипичные сценарии | Междоменные темы, большие работы |

### Файлы: 7 штук (было 6)

Добавлен `implementation_recipes.md` — 10 готовых рецептов с полным кодом. Это ключевой файл для качества: без него Claude Code будет создавать графики «как попало», с ним — профессионального уровня.
